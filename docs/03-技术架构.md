# xiaochu2 技术架构设计

> 引擎：PixiJS 7 + TypeScript + Vite，复用 game2D_huahua 已验证的微信/抖音真机适配层。

## 1. 为什么换引擎

xiao_chu 用原生 Canvas 2D：

- 战斗场景每帧全屏 `clearRect` + 全量重绘，CPU 绘制是瓶颈
- 棋盘动画每帧 `createLinearGradient`，对象分配压力大
- 靠 `battleMemoryGuard` 限流粒子/飘字只是兜底

PixiJS 7 走 WebGL：Sprite 场景图 + GPU 合批，珠子/特效/飘字是常驻显示对象，
每帧只改 transform 不重绘，从根上解决卡顿。

## 2. 技术选型

| 项 | 选择 | 理由 |
|---|---|---|
| 渲染 | PixiJS 7.3.x（不用 v8） | v8 的 unsafe-eval / OffscreenCanvas / DOMAdapter 在小游戏环境问题多，v7 有 huahua 真机验证 |
| 语言 | TypeScript（strict） | 数值表 / 公式签名类型约束 |
| 构建 | Vite 6 + IIFE 单文件输出 | 复用 huahua 配置：`@pixi/*` dedupe + unsafe-eval 后处理插件 |
| 补间 | 自研 TweenManager（移植） | 零依赖，宠物攻击动效够用 |
| 骨骼动画 | 暂不引入 spine | 首版用 Sprite 位移 + AnimatedSprite 帧动画，留接口 |

## 3. 启动链路

```
minigame/game.js
  → require('./pixi-adapter/index')   # DOM/window/canvas/touch/XHR 全局 shim
  → require('./game-bundle.js')       # Vite IIFE 构建产物
      main.ts 第一行 import pixiUnsafeEvalPatch  # ShaderSystem/ADAPTER/Texture.WHITE patch
      Game.init(GameGlobal.canvas)               # 750 设计宽、DPR、Renderer 三级降级
      TextureCache.preload(...)
      SceneManager.switchTo('title')
```

## 5. 渲染分层

```
Game.stage (sortableChildren)
├── Scene.container
│   ├── 背景层
│   ├── 敌人层（受击动效）
│   ├── 棋盘层（_cellsLayer 格底 / _beadsLayer 珠子，珠子走对象池）
│   ├── 队伍栏层（宠物头像，攻击时冲刺位移）
│   └── 特效层（ParticleContainer 粒子 + 飘字池）
└── OverlayManager.container (zIndex 10000, 全局弹窗)
```

## 6. 性能策略（转珠场景专项）

| 手段 | 说明 |
|---|---|
| 珠子对象池 | 36 颗珠 + 消除/下落复用 Sprite，不销毁重建 |
| ParticleContainer | 消除爆裂粒子，按机型分 lite/full 两档 |
| 纹理图集 | 珠子/战斗 UI 打 Spritesheet，减少 draw call 与解码内存 |
| 降帧 | 浏览器：`Game.setMaxFPS`；**小游戏真机禁止 Pixi maxFPS**（见踩坑记录） |
| 纹理缓存 | TextureCache Map 缓存 + inflight 去重 + 失败重试 |
| 内存 | 场景切换 `destroy({children:true})`，Tween 目标销毁检测 |

## 7. 多平台

- `pixi-adapter/platform.js`：`wx` / `tt` API 底层探测
- `PlatformService.ts`：业务层封装（存储/广告/分享/生命周期）
- 微信与抖音各自维护 `game.json` / 工程配置，TS 层零分支
